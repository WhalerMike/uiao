ORGPATH GOVERNANCE SERIES — PART EIGHT SUPPLEMENT

OrgPath with Third-Party DDI Platforms

Infoblox and BlueCat Integration Supplement

Revision 1.0 \| May 2026 \| Companion to: *OrgPath Part Eight — Azure-Native DDI with Windows Server IPAM and Azure DNS*

This supplement extends Part Eight of the OrgPath governance series. Part Eight established the Azure-native DDI pattern: Windows Server IPAM as the on-premises IP address management interface, Azure DNS Private Zones as the cloud-hosted authoritative namespace, and the Azure DNS Private Resolver as the conditional forwarder fabric that bridges the two. That architecture is the long-term target state. What Part Eight acknowledged but did not detail is the reality that a significant number of enterprise networks already operate a third-party DDI platform — Infoblox NIOS, Infoblox BloxOne DDI, or BlueCat Address Manager — and will continue to operate it throughout a multi-year hybrid transition. This supplement addresses those environments directly.

The document is organized in five chapters. The first establishes why third-party DDI platforms exist as a rational architectural choice during the hybrid period and what role they occupy within the broader OrgPath governance model. The second covers the Infoblox integration in detail, addressing both NIOS Grid deployments and the BloxOne cloud-native variant. The third covers BlueCat Address Manager. The fourth describes how OrgPath keeps both a third-party DDI platform and Azure DNS coherent during the Parallel migration phase when both are simultaneously active. The fifth establishes the exit criteria and operational procedure for retiring the DDI platform when Azure-native constructs are ready to take full ownership.

Every PowerShell script in this document is complete and fully annotated. Environment-specific values — hostnames, credential objects, resource group names, subscription identifiers, and naming conventions — are marked in comments so they can be found and substituted without reading the surrounding code. The scripts are designed to be idempotent: running them repeatedly against the same environment produces the same result without creating duplicate objects or overwriting values that are already correct.

**CHAPTER ONE**

The Case for Third-Party DDI During the Hybrid Transition

The most direct way to describe what Infoblox and BlueCat provide that Azure-native tooling does not is to describe the view from the operations center during an active hybrid migration. On one screen, an administrator has the Azure portal open, displaying the DNS Private Zones blade for the hub subscription — a list of zones, some populated with records, some still empty stubs waiting for their on-premises counterparts to be migrated. On a second screen, the same administrator has a Windows Server IPAM console open, displaying the DHCP scopes and DNS zones managed by the on-premises domain controllers. These are two distinct management planes. They share no query interface, no audit trail, and no change workflow. A query against one does not produce results from the other. An audit log entry in Azure Monitor tells the administrator what changed in Azure DNS; a Windows Event Log entry on the IPAM server tells the administrator what changed on-premises. The two logs are not correlated, and there is no native mechanism to correlate them.

Azure DNS Private Zones manages cloud-hosted private namespaces with precision and at scale. The Windows Server IPAM role manages the on-premises DHCP and DNS estate through a Microsoft-native interface that integrates cleanly with Active Directory and with the DHCP and DNS roles on Windows Server. What neither provides is a single unified management plane that spans both surfaces simultaneously — a console where an administrator can query every subnet, every DHCP scope, every DNS zone, and every IP address allocation across both on-premises and Azure, side by side, using a single query language, producing a single audit trail, and enforcing a single change workflow. This is the capability gap that third-party DDI platforms fill, and it is a gap that becomes operationally significant the moment an organization begins moving workloads between environments at any meaningful pace.

The consequences of managing two uncoordinated address spaces during a hybrid transition are concrete and predictable. IP address conflicts between on-premises ranges and cloud-allocated virtual network address spaces cause connectivity failures that are difficult to diagnose because neither side reports the conflict — each system believes its allocations are authoritative, and the failure manifests only when a packet attempts to route to an address that is claimed by both environments. DNS mismatches between zones that are still authoritative on on-premises domain controllers and zones that are being built out in Azure DNS Private Zones cause name resolution failures for specific client populations: Azure-hosted workloads resolve from Azure DNS and may see an incomplete or divergent zone, while on-premises clients resolve from domain controller DNS and see the original zone in full. DHCP scopes that are partially migrated — some addresses managed on-premises, others implicitly allocated by Azure Virtual Network — produce stale leases on both sides, with on-premises DHCP servers offering addresses that Azure may have already assigned to a virtual machine NIC and Azure offering address ranges that on-premises DHCP scopes have not been told to exclude.

A third-party DDI platform — Infoblox NIOS, Infoblox BloxOne, or BlueCat Address Manager — provides the unified view that prevents these failures. All three platforms are designed specifically for the problem of managing heterogeneous network address and DNS infrastructure from a single management plane. They maintain a master record of every allocated subnet, every DHCP scope and its lease history, and every DNS zone and its record set, regardless of where those objects physically reside. An administrator working in the Infoblox Grid Manager or the BlueCat Address Manager interface can query the entire IP space with a single search and see, in a single result set, subnets allocated to on-premises VLANs and subnets assigned to Azure virtual networks. A change to a DNS zone travels through the DDI platform's change workflow — approval, audit, rollback — before being committed to the authoritative name server, whether that name server is an on-premises domain controller DNS or a BlueCat-managed DNS server. The unified audit trail that the platform produces satisfies operational governance requirements that two disconnected audit logs do not.

The Architecture This Document Describes

The integration architecture described in this document positions the third-party DDI platform as the system of record for IP address management and DNS authority during the hybrid transition period. This is not a parallel or shadow system — it is the primary management interface through which network and DNS changes are made, reviewed, and approved. The OrgPath governance pipeline extends to include the DDI platform's REST API as a target alongside the Entra ID directory, the Intune endpoint management plane, and Azure Resource Manager. When an OrgTree node is created, modified, or retired in the governance repository, the pipeline propagates the change to every downstream system: Entra ID receives updated dynamic group membership rules, Intune receives updated compliance policy assignments, Azure Resource Manager receives updated resource tags on Arc-enrolled servers and Azure workloads, and the DDI platform receives updated extensible attributes on the DNS zones and IP networks that correspond to the changed node. The organizational context — the OrgPath identifier, the node ID, the management tier, the migration phase — is stamped on every object in every system simultaneously by the same pipeline run.

The transition from this architecture to the pure Azure-native state described in Part Eight is governed by a migration phase tracking mechanism that this document describes in detail. Each DNS zone and IP network object in the DDI platform carries a MigrationPhase attribute that moves through four states: Assessment, Migration, Parallel, and Complete. Assessment means the object exists in the DDI platform and has been tagged with OrgPath attributes but has no corresponding Azure DNS Private Zone. Migration means an Azure DNS Private Zone has been created but is not yet serving live resolution traffic. Parallel means both the DDI platform zone and the Azure DNS Private Zone are active and serving resolution for their respective client populations — this is the steady-state during the main body of the transition. Complete means the Azure DNS Private Zone is the sole authoritative source, all clients have been redirected, and the DDI platform zone is retired. When every object in the DDI platform carries a MigrationPhase of Complete, the decommissioning criteria described in Chapter Five are met and the DDI platform can be retired.

**CHAPTER TWO**

Infoblox — Extensible Attributes as the OrgPath Bridge

Infoblox offers two deployment models that are relevant to this integration, and the choice between them is typically determined by what an organization already has in place rather than by a fresh architectural decision. Infoblox NIOS is the on-premises Grid deployment: a set of physical or virtual appliances — Grid Members and a Grid Master — managed through the NIOS web interface and through the WAPI REST API. NIOS has been the dominant enterprise DDI platform for more than a decade and the majority of large enterprise networks that operate an Infoblox deployment are running NIOS in some form. Infoblox BloxOne DDI is the cloud-native SaaS successor, managed through the Infoblox Cloud Services Portal at csp.infoblox.com and delivered as a managed service. BloxOne can coexist with a NIOS Grid or operate independently; organizations migrating from NIOS to BloxOne may run both simultaneously during the transition.

Both platforms expose a REST API surface for network, zone, and object management, and the OrgPath integration pattern is identical in concept on both. NIOS WAPI calls are directed to the base URL https://{gridmaster-fqdn}/wapi/v2.12/, where v2.12 corresponds to NIOS 8.x builds; earlier NIOS versions use earlier WAPI versions and the version component in the URL should be adjusted accordingly. BloxOne API calls are directed to https://csp.infoblox.com/api/ddi/v1/ and authenticated with a Bearer token rather than HTTP Basic credentials. Beyond the authentication model and the base URL, the structural intent of the API calls is the same: network objects, DNS zones, and DHCP scope objects carry OrgPath values as metadata — Extensible Attributes in the NIOS model, tags in the BloxOne model — and the governance pipeline reads and writes those metadata values through the respective API endpoints. The downstream effect, from the OrgPath model's perspective, is identical: every object in the DDI system expresses the same organizational context that every object in Entra ID, Intune, and Azure Resource Manager expresses.

Infoblox Extensible Attributes — Structure and Definition

Extensible Attributes are custom metadata fields that Infoblox administrators define globally in the NIOS system and then assign to any object type — networks, zones, hosts, fixed addresses, DHCP scopes, and DNS records. An EA definition has a name, a data type, and optionally a list of allowed values for Enum-type attributes. Once defined, an EA is available across the entire Grid and can be set on any supported object. The inheritance flag on an EA definition controls whether child objects — subnets within a network container, zones within a parent zone — automatically inherit the EA value from their parent unless explicitly overridden. This inheritance behavior is useful for OrgPath integration because it means that setting the OrgPath EA on a top-level network container automatically propagates the value to all subnets within it, without requiring the governance pipeline to enumerate and tag every child object individually.

For OrgPath integration, four EA definitions are required in the NIOS Grid. OrgPath is a String EA carrying the full hierarchy path from the OrgTree root to the governing node, expressed in the standard OrgPath path notation established in earlier parts of this series — for example, /root/operations/infrastructure/network. The inheritable flag is set on this EA so that child objects within a tagged container inherit the path unless a more specific OrgPath value is assigned at a lower level. OrgNodeId is a String EA carrying the leaf node identifier from the OrgTree registry — the short, stable identifier used as the primary key for lookups in the governance pipeline. It is also marked inheritable. OrgTier is an Enum EA with allowed values of Tier0, Tier1, Tier2, and Unclassified, representing the management tier classification applied to the node in the OrgTree registry; the inheritable flag is set here as well so that all objects within a Tier0 network container are treated as Tier0 by default. MigrationPhase is an Enum EA with values of Assessment, Migration, Parallel, and Complete; this attribute is not inheritable because migration phase is a property of individual zones and networks, not a property that should cascade from a container to every child object automatically.

These four EA definitions are created once as a setup step, before any governance pipeline synchronization runs against the Grid. The following PowerShell function performs this registration via the NIOS WAPI REST API. It is idempotent: it checks for the existence of each EA definition before attempting to create it, so it can be run safely against a Grid where some or all of the definitions already exist.

The NIOS Synchronization Pipeline Script

With the EA definitions in place, the governance pipeline can begin stamping OrgPath values on network and zone objects in the Grid. The synchronization script below is the NIOS counterpart to the Sync-OrgPathDnsZones.ps1 script described in Part Eight for Azure DNS Private Zones. It reads the OrgTree registry JSON file that is checked out from the governance repository at the start of each pipeline run, iterates over every active node, and for each node that is flagged as a DNS boundary or carries a network CIDR in its metadata, either creates the corresponding NIOS object or updates the Extensible Attributes on an existing object. The script is triggered on OrgTree commits as part of the governance propagation pipeline and runs on the automation server — a Windows Server host that has network access to the Grid Master WAPI endpoint and the necessary credentials stored in the Windows Credential Manager or a secrets vault.

Infoblox BloxOne DDI — Bearer Token Authentication and Tag Structure

BloxOne DDI uses a fundamentally different authentication model and a different metadata structure than NIOS, even though the conceptual purpose of the integration is identical. Where NIOS uses HTTP Basic authentication against the Grid Master, BloxOne uses Bearer token authentication: an API key is generated in the Cloud Services Portal under Administration, then API Keys, then Create API Key. That key is presented in the Authorization header of every API call as Token {api-key-value}. The key should never appear in a script file; it should be stored in the Windows Credential Manager, a secrets vault such as Azure Key Vault, or the pipeline runtime's secure variable storage, and retrieved into a PowerShell variable at runtime.

Where NIOS carries metadata in the extensible_attributes object with the value nested as { "value": "..." }, BloxOne carries metadata in a flat tags object where each key maps directly to a string value. The BloxOne resource paths also differ from NIOS: subnets are managed at /api/ddi/v1/ipam/subnet, authoritative zones at /api/ddi/v1/dns/auth_zone, and DNS records at /api/ddi/v1/dns/record. The following function demonstrates the BloxOne authentication pattern and the zone creation call with embedded OrgPath tags. It is structured as a function rather than a script so it can be called from the main governance pipeline orchestrator, which handles the OrgTree registry iteration and passes individual node values to platform-specific functions like this one.

**CHAPTER THREE**

BlueCat — OrgPath via User-Defined Fields and the REST API

BlueCat Address Manager organizes its data model around a hierarchy of Configurations, each of which contains all the IP address space and DNS infrastructure under its governance. Within a Configuration, IP address space is organized into a Block and Network hierarchy — a Block is a large address range, and a Network is a specific subnet within a Block. DNS infrastructure is organized into a View and Zone hierarchy — a View is a logical DNS namespace (used for split-horizon DNS), and Zones exist within Views. This hierarchical structure maps naturally to the OrgTree model: an OrgTree node that governs a specific DNS namespace maps to a Zone in a BlueCat View, and an OrgTree node that governs a specific IP address range maps to a Network within a BlueCat Block.

Every object in the BlueCat model supports User-Defined Fields — metadata fields that administrators define in the BAM Administration interface under User-Defined Fields and then attach to specific object types. UDFs are the BlueCat equivalent of Infoblox Extensible Attributes and serve the same purpose in the OrgPath integration: they carry OrgPath governance values on each object so that the same organizational context readable in Entra ID, Intune, and Azure Resource Manager is also readable by querying the BlueCat API. Before running the synchronization script in this chapter, an administrator must pre-define four UDFs in the BAM Administration interface: OrgPath (Text type), OrgNodeId (Text type), OrgTier (Text type), and MigrationPhase (Text type). Unlike Infoblox EAs, BlueCat does not offer an Enum UDF type in all versions; Text is used for all four fields and the governance pipeline enforces value constraints, as with the BloxOne integration described in the previous chapter.

The BlueCat REST API — Session Model and Key Endpoints

BlueCat Address Manager exposes a REST API at https://{bam-host}/Services/REST/v1/ that supports HTTP Basic authentication through a session model. A login call authenticated with Basic credentials returns a session token string in the form BAMAuthToken: {token-value}; that token is then passed as the Authorization header value in all subsequent calls within the session. The session must be explicitly closed with a logout call when the script completes. Leaving sessions open is not a security risk in itself, since the token expires, but it does consume BAM server-side session resources and should be avoided in automation scripts that run frequently.

BlueCat has introduced a v2 REST API at https://{bam-host}/api/v2/ in more recent BAM versions. The v2 API is more RESTful in design — it uses standard HTTP verbs against resource-oriented URLs rather than the action-based URLs of the v1 API — and it will eventually supersede the v1 API. However, because many enterprise BAM deployments are running versions that do not yet fully support the v2 API, and because the v1 API is stable and well-documented across all current BAM versions, the integration script in this chapter uses the v1 API. The conceptual structure — authenticate, retrieve configuration, look up objects, write UDF values — is identical in both API versions.

The key v1 API endpoints for OrgPath integration are as follows. The /login endpoint performs session establishment. The /getEntityByName endpoint retrieves an object by name within a parent entity — this is the primary lookup mechanism for finding Configurations, Views, Zones, and Networks by name. The /getEntities endpoint retrieves all child objects of a given type under a parent entity — used to enumerate Views within a Configuration. The /updateObject endpoint updates the properties and UDF values of an existing object by its internal numeric ID. The /addZone endpoint creates a new DNS zone as a child of a View. The /logout endpoint closes the session.

**CHAPTER FOUR**

Running Two Systems in Parallel — How OrgPath Keeps Them Coherent

The Parallel migration phase is the most operationally demanding period in the hybrid transition. It is the period during which both the third-party DDI platform and Azure DNS Private Zones are simultaneously active and serving live DNS resolution traffic — the former for on-premises clients, the latter for Azure-hosted workloads. A zone in the Parallel phase exists in two places: as an authoritative zone in the NIOS Grid or the BlueCat DNS View, with its records managed through the DDI platform's change workflow, and as an Azure DNS Private Zone in the hub subscription, with its records propagated from the DDI platform or maintained in parallel by the Azure DNS team. The OrgPath governance pipeline is the single agent that writes organizational context to both simultaneously, and the MigrationPhase extensible attribute is the shared state variable that tells both systems — and the operators managing them — exactly where in the transition sequence each zone currently sits.

Understanding the state machine that governs zone lifecycle during the Parallel phase requires understanding what each state transition represents in concrete operational terms. A zone enters the Assessment state when the OrgPath synchronization script first creates it in the DDI platform or first stamps OrgPath EAs on an existing zone that predates the OrgPath integration. Assessment means that the zone is tagged but no migration work has begun — no Azure DNS Private Zone exists, no records have been replicated, and the zone is fully authoritative in the DDI platform. A zone moves to Migration state when the governance team has created an Azure DNS Private Zone with a matching FQDN and stamped it with the same OrgPath values, but has not yet pointed any clients at the Azure zone — it exists as an empty or partially populated zone in Azure, not yet serving live traffic. A zone enters the Parallel state when the Azure DNS Private Zone is active and receiving resolution queries from Azure-hosted workloads, while the NIOS or BAM zone remains the authoritative source for on-premises clients. In this state, any record change must be applied to both zones — to the DDI platform zone through its normal change workflow, and to the Azure zone through the Azure DNS API or the governance pipeline's Azure synchronization component.

The transition from Parallel to Complete is the most consequential state change in the entire migration sequence, because it is the point at which on-premises clients are redirected from the DDI platform's name servers to the Azure DNS Private Resolver. The governance pipeline enforces this transition through a prerequisite check: a zone's MigrationPhase attribute cannot be updated to Complete unless the pipeline can confirm, through live API calls, that an Azure DNS Private Zone with the matching FQDN exists, carries a matching OrgPath tag, is linked to the hub virtual network, and is accessible from the Azure DNS Private Resolver's inbound endpoint. If any of these conditions are not met, the pipeline blocks the state transition and writes a blocking reason to the governance log. This is the same Decommissioning Readiness Score pattern introduced in Part Eight, applied at the individual zone level.

The Parallel-Phase Record Synchronization Problem

One operational challenge specific to the Parallel phase is record synchronization: when a new DNS record is created in the DDI platform zone — because a new on-premises server is being deployed, for example — that record must also be created in the Azure DNS Private Zone so that Azure-hosted clients can resolve it without waiting for the next scheduled synchronization run. The governance pipeline addresses this through a change-feed subscription pattern. Both Infoblox NIOS and BlueCat BAM support outbound notifications or webhook-style integrations that fire when a record is created or modified. In NIOS, the Grid's Outbound API notification feature can be configured to POST a JSON payload to the pipeline webhook URL whenever a record change is committed to a zone in the Parallel migration phase. The pipeline webhook receiver, running as an Azure Function or as a Logic App, receives the notification, extracts the record data, and issues an Azure DNS record creation or update call through the Az.PrivateDns module. The record is in Azure DNS within seconds of being committed in NIOS, without any polling delay.

In environments where the DDI platform's outbound notification feature is not available — older NIOS versions, certain BlueCat configurations, or network policy restrictions that prevent the DDI appliance from making outbound HTTP calls to cloud endpoints — the pipeline falls back to a polling approach: a scheduled task runs every fifteen minutes, queries the DDI platform for zone records modified in the last fifteen minutes, and replays any changes to the Azure DNS Private Zone. The polling interval should be chosen based on the organization's tolerance for DNS replication lag; fifteen minutes is conservative and appropriate for most workloads. Latency-sensitive workloads that require near-real-time record propagation should be migrated to the Complete phase — and therefore out of the Parallel split-brain configuration — before the on-premises host publishing those records is migrated to Azure.

**CHAPTER FIVE**

The Exit Criteria — When Azure Native Takes Full Ownership

Retiring the third-party DDI platform is not a single event — it is the conclusion of a sequence of verifiable conditions, each of which must be confirmed before the next step proceeds. The platform has been the system of record for the organization's DNS and IP address estate, in some cases for a decade or more, and its retirement cannot be treated as a routine infrastructure decommission. The following exit criteria define the state that the environment must reach before the retirement sequence can begin. They are organized around the same Decommissioning Readiness Score framework introduced in Part Eight, extended here to account for the additional surface area introduced by the third-party DDI platform.

The first criterion is zone completion. Every DNS zone that was authoritative in the NIOS Grid or the BlueCat DNS View must carry a MigrationPhase value of Complete, and must have a corresponding Azure DNS Private Zone tagged with a matching OrgPath value, linked to the hub virtual network, and accessible from the Azure DNS Private Resolver's inbound endpoint. This is a hard requirement: the existence of even a single zone in the Assessment, Migration, or Parallel state means that some DNS resolution is still being served from the DDI platform, and retiring it would cause resolution failures for clients that depend on those zones. The reconciliation script in Chapter Four is the mechanism that verifies this criterion: the report must show zero NIOS-only zones, zero Azure-only zones without a corresponding Complete-phase NIOS entry, and zero phase mismatches before the first criterion is considered met.

The second criterion is DHCP scope migration. Every DHCP scope that was managed through the DDI platform must be accounted for in one of two ways. Either the address range that the scope served has been absorbed into Azure Virtual Network address space, meaning that all hosts that formerly received dynamic addresses from the on-premises DHCP scope are now Azure-hosted and receive their address assignments through Azure's built-in DHCP mechanism. Or the scope has been migrated to a Windows Server DHCP server managed through the Windows Server IPAM role as described in Part Eight, with the DHCP failover relationship between the migrated server and its partner verified and the DDI platform's scope record marked as decommissioned in the governance log. No DHCP scope that was active in the DDI platform can be left without an explicit migration disposition in the governance record. A scope that is simply deleted from the DDI platform without being migrated represents an address management gap that may not surface immediately but will cause lease conflicts or addressing failures later.

The third criterion is forwarder removal. All conditional forwarding rules on the Azure DNS Private Resolver Forwarding Ruleset that were created to forward queries for on-premises zones to the NIOS or BAM name server IP addresses must have been removed. The presence of a forwarding rule for a zone whose NIOS or BAM counterpart has been marked Complete indicates a configuration inconsistency: the zone is fully hosted in Azure DNS, but the resolver is still forwarding queries for it to an on-premises name server that is no longer the authoritative source. In an active system this would cause resolution failures for those queries. Removing the forwarding rule for each zone as its MigrationPhase is advanced to Complete — rather than waiting until the end of the entire migration — is the recommended approach, both because it validates the Complete state in real time and because it ensures that the resolver configuration converges to its final state incrementally rather than all at once. The final check before retirement is simply confirming that no forwarding rules remain that reference any of the DDI platform's name server addresses.

The fourth criterion is the final reconciliation. The reconciliation script from Chapter Four is run one final time, and its output must show zero entries in the NIOS-only, Azure-only, and phase mismatch categories. The fully synchronized count must equal the total number of zones that were previously managed in the DDI platform, confirming that every zone has been accounted for in Azure DNS and that no zone has been lost during the migration. This report is archived in the governance repository as the definitive migration completion record.

The Operational Retirement Procedure

Once all four exit criteria have been verified, the retirement procedure begins. The first step is placing the DDI platform in read-only mode. In Infoblox NIOS, this is accomplished through the Grid's access control system: the service account used by the governance pipeline and the day-to-day DNS administration accounts are demoted to read-only roles, removing the ability to create, modify, or delete any DNS record, zone, or network object in the Grid. The Grid Master interface remains accessible for monitoring and audit log queries but accepts no write operations. In BlueCat Address Manager, the equivalent is achieved through the BAM access control mechanism, setting all non-administrative user accounts to read-only access at the Configuration level. The network operations team is notified of the read-only transition and advised that any DNS or IPAM changes that arise during the monitoring window must be made through Azure-native tooling — the Azure portal, the Az.PrivateDns module, or the governance pipeline — rather than through the DDI platform interface.

The monitoring window is thirty calendar days from the date of the read-only transition. During this period, DNS query logs from the Azure DNS Private Resolver are the primary monitoring artifact. The resolver logs — streamed to an Azure Monitor Log Analytics workspace through the diagnostic settings on the resolver resource — are queried daily to verify that no queries are being forwarded to the DDI platform's name server IP addresses. A query that reaches the resolver and matches no local Azure DNS Private Zone would normally generate a SERVFAIL or NXDOMAIN response; if such responses are observed at a rate higher than baseline during the monitoring window, the resolution path for those queries must be traced to determine whether they represent a forwarding rule that was missed during the cleanup step or a client configuration that is bypassing the resolver entirely and querying the old DDI platform name servers directly.

If the thirty-day monitoring window passes without any forwarded queries to the DDI platform's name servers, the Grid Master or BAM appliances are decommissioned following the organization's standard server decommission procedure. The virtual machines or physical appliances are powered off, snapshots are taken for archive purposes, and the licenses are returned to the vendor. The governance repository receives a final commit recording the retirement date in ISO 8601 format, the path to the final reconciliation report, the names of the engineers who performed the verification steps, and the names of the approvers who authorized the retirement. This commit message becomes the permanent record of the DDI platform's lifecycle within the OrgPath governance model.

Every OrgTree node whose MigrationPhase field has not already been updated to Complete during the zone-by-zone migration sequence is updated in this final commit. The MigrationPhase field in the OrgTree registry, from this point forward, is a historical attribute — it records the migration state at the time of retirement, not an active governance control. The active governance controls for DNS and IP address management are now entirely the Azure DNS Private Zones, the Azure Virtual Network address space assignments, and the Azure DNS Private Resolver forwarding configuration, all of which carry OrgPath tags set and maintained by the same governance pipeline that formerly wrote to the DDI platform's REST API. The pipeline target list shrinks by one entry: the DDI platform's API endpoint is removed, and the pipeline routes all DNS and IPAM governance operations to Azure-native APIs.

What the DDI Platform Leaves Behind

The third-party DDI platform's role in the OrgPath architecture was always defined by its temporality. It was a bridge — a system of record that maintained visibility and operational control during the period when neither the on-premises Active Directory-integrated DNS and DHCP nor the Azure-native DNS and virtual network address management was sufficient on its own to govern the full network estate. The bridge served its purpose: it prevented IP address conflicts by maintaining a unified address inventory, it prevented DNS split-brain failures by providing a single authoritative view of the zone inventory during the transition, and it provided the audit trail and change workflow that the organization's operational governance requirements demanded during the period when two separate environments were being managed simultaneously.

OrgPath was the thread that ran through the bridge. The same organizational context — the same node identifiers, the same hierarchy paths, the same tier classifications — that was encoded on Entra ID user objects, Intune device objects, Arc resource tags, and Microsoft Defender device groups was also encoded on NIOS zones and networks, on BlueCat zones and IP networks, and on Azure DNS Private Zones. Every system that touched a governed resource could answer the same question — what organizational context does this object belong to? — through its own native query interface, without any cross-system lookup. This is the value that the OrgPath model provides that no individual system provides alone.

When the bridge is retired, the organizational context does not retire with it. It has already been propagated — zone by zone, network by network, device by device — to every Azure-native construct that carries organizational identity forward. The DDI platform leaves, and what remains is an environment where every DNS zone, every virtual network subnet, every resource tag, every device object, and every identity object carries the same coherent organizational context, expressed entirely through Azure-native mechanisms that require no additional platform to maintain. The OrgPath model has not changed. The substrate it runs on has converged, at last, to a single system — and the governance pipeline that enforces it runs lighter, faster, and with fewer moving parts than at any point during the transition. That convergence is what the entire hybrid migration was building toward, and the retirement of the DDI platform is the operational marker that confirms it has been reached.
