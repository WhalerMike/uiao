# UIAO Adapter Developer Training Program

This document defines the training curriculum for developers building
UIAO adapters. It covers the architecture, patterns, tooling, and
governance requirements that every adapter developer must understand.

---

## 1. Program Overview

### 1.1 Audience

- New developers joining the UIAO adapter team
- Existing developers building a new adapter for the first time
- Security engineers reviewing adapter implementations
- Auditors assessing adapter compliance

### 1.2 Learning Objectives

After completing this program, participants will be able to:

1. Explain the dual-axis adapter taxonomy (operational + doctrinal)
2. Implement a new adapter subclassing `DatabaseAdapterBase`
3. Write conformance-passing unit and integration tests
4. Generate OSCAL artifacts (SAR, POA&M, SSP) from adapter output
5. Follow the canon-consumer rule and three-stage lifecycle
6. Debug adapter conformance failures using the CI gate

### 1.3 Prerequisites

- Python 3.11+ proficiency
- Familiarity with pytest
- Basic understanding of NIST SP 800-53 controls
- Git workflow (branching, PRs, CI)

## 2. Module 1: Architecture Foundations (2 hours)

### 2.1 Reading

| Document | Location | Focus |
|----------|----------|-------|
| ARCHITECTURE.md §3.2 | uiao-core | Dual-axis taxonomy |
| UIAO_003 §4.1–§4.7 | uiao-core/canon | Five adapter classes |
| Adapter Doctrine | UIAO_003 §4.1 | SSOT + Identity + Security |
| DatabaseAdapterBase | impl/src/uiao_impl/adapters/database_base.py | 7 canonical domains |

### 2.2 Key Concepts

**Dual-axis taxonomy:**
- Operational class (`class`): `modernization` (change-making) vs `conformance` (read-only)
- Doctrinal mission class (`mission-class`): identity, telemetry, policy, enforcement, integration

**7 Canonical Domains:**
1. Connection & Identity → `connect() → ConnectionProvenance`
2. Schema Discovery → `discover_schema() → SchemaMappingObject`
3. Query Normalization → `execute_query() → QueryProvenance`
4. Data Normalization → `normalize() → ClaimSet`
5. Drift Detection → `detect_drift() → DriftReport`
6. Evidence Packaging → `collect_evidence() → EvidenceObject`
7. Convenience → `collect_and_align() → Dict`

**Canon invariants (MUST hold for every adapter):**
- `gcc-boundary: gcc-moderate`
- `ssot-mutation: never`
- `certificate-anchored: true`
- `object-identity-only: true`

### 2.3 Exercise

Read `servicenow_adapter.py` end-to-end. For each of the 7 methods,
identify: (a) what vendor data it consumes, (b) what canonical type it
returns, (c) how it computes the provenance hash.

## 3. Module 2: Building Your First Adapter (4 hours)

### 3.1 The Three-Stage Lifecycle

| Stage | Repo | What you do | Gate |
|-------|------|-------------|------|
| 1. Canon | uiao-core | Add registry entry (`status: reserved`) | Schema validates |
| 2. Docs | uiao-docs | Scaffolds auto-generated by canon-sync (or manual sync_canon.py) | sync_canon.py --check-only = 0 |
| 3. Impl | uiao-impl | Subclass DatabaseAdapterBase + tests | 30/30 conformance + CI green |

### 3.2 Step-by-Step: Building a "Hello World" Adapter

```python
# src/uiao_impl/adapters/hello_adapter.py
from .database_base import (
    ClaimObject, ClaimSet, ConnectionProvenance, DatabaseAdapterBase,
    DriftReport, EvidenceObject, QueryProvenance, SchemaMappingObject,
)

class HelloAdapter(DatabaseAdapterBase):
    ADAPTER_ID: str = "hello"

    def __init__(self, config=None):
        super().__init__(config or {})

    def connect(self) -> ConnectionProvenance:
        return ConnectionProvenance(
            identity=f"hello:{self._config.get('name', 'world')}",
            auth_method="none",
            endpoint="https://hello.local",
            tls_version=None,
            mtls_enabled=False,
            timestamp=self._now(),
        )

    def discover_schema(self) -> SchemaMappingObject:
        return SchemaMappingObject(
            vendor_schema={"greeting": "string", "target": "string"},
            canonical_schema={"identity": "hello:<target>"},
            mapping_rules={"target": "identity suffix"},
            unmapped_fields=["metadata"],
            version_hash=self._hash({"vendor": {"greeting": "string"}}),
        )

    def execute_query(self, q):
        return QueryProvenance(
            canonical_query=q,
            vendor_query=f"SELECT * FROM greetings WHERE target='{q.get('from', '*')}'",
            execution_plan_hash=self._hash(q),
            row_count=0,
            timestamp=self._now(),
        )

    def normalize(self, raw_rows):
        claims = []
        for row in raw_rows:
            claims.append(ClaimObject(
                claim_id=f"hello:{row.get('target', 'unknown')}",
                entity=f"hello:{row.get('target', 'unknown')}",
                fields={"greeting": row.get("greeting", ""), "target": row.get("target", "")},
                source=self.ADAPTER_ID,
                provenance_hash=self._hash(row),
            ))
        return ClaimSet(claims=claims, source_reference="hello-world")

    def detect_drift(self):
        return DriftReport(
            drift_type="hello-greeting",
            severity="info",
            first_observed=self._now(),
            last_observed=self._now(),
            details={"adapter": self.ADAPTER_ID, "message": "No drift in greetings."},
        )

    def collect_and_align(self):
        return {"vendor": "Hello World", "adapter_id": self.ADAPTER_ID,
                "claims": self.normalize([]).to_dict(),
                "metadata": {"last_collected": self._now().isoformat()}}
```

### 3.3 Exercise

1. Create `hello_adapter.py` following the template above
2. Register it in `__init__.py` and `test_adapters.py`
3. Run `conformance_check.py --adapter=hello` → verify 30/30
4. Write 3 behavioral tests against a fixture file
5. Generate a SAR using `build_adapter_bundle() → build_sar()`

## 4. Module 3: Vendor-Specific Depth (3 hours)

### 4.1 Real Parser Pattern

Study `terraform_parser.py` as the reference implementation:

| Function | What it does | Pattern |
|----------|-------------|---------|
| `parse_tfstate()` | JSON → flat resource list | Load → iterate → normalize |
| `parse_plan_json()` | Plan JSON → change list with severity | Action mapping + diff computation |
| `parse_hcl()` | HCL2 → desired-state list | External library (python-hcl2) |
| `three_way_diff()` | Compare three sources | Set operations on keyed resources |

### 4.2 Vendor Edge Cases

Every real adapter must handle:

| Edge case | Example | Pattern |
|-----------|---------|---------|
| Rate limiting | Graph API 429 | `from .retry import with_retry` |
| Pagination | ServiceNow `sysparm_offset` | Loop until empty page |
| Auth token refresh | CyberArk session expiry | Token cache + refresh logic |
| Large payloads | 10K+ resources in state | Streaming parse, batch normalize |
| Missing fields | Optional vendor attributes | `.get()` with defaults, never KeyError |
| Encoding | XML namespaces, UTF-8 BOM | Explicit encoding parameter |

### 4.3 Exercise

Pick one of the existing adapters (M365, Palo Alto, or VulnScan) and:
1. Read the parser module
2. Identify one edge case not currently handled
3. Write a test that exposes the edge case
4. Implement the fix

## 5. Module 4: OSCAL Pipeline (2 hours)

### 5.1 The Bridge: adapter_to_oscal.py

```
adapter output → claims_to_ir_evidence() → IR Evidence
                → drift_to_ir_states() → IR DriftState
                → build_adapter_bundle() → EvidenceBundle
                → build_sar() / build_adapter_poam() / build_adapter_ssp()
                → OSCAL JSON
```

### 5.2 Three Artifact Types

| Artifact | Input | Output | Use |
|----------|-------|--------|-----|
| SAR | ClaimSet | Assessment Results | "What did we find?" |
| POA&M | DriftReport | Plan of Action | "What needs fixing?" |
| SSP | ClaimSet | Security Plan | "How are controls implemented?" |

### 5.3 Exercise

Generate all three OSCAL artifacts from the Terraform adapter fixture:

```python
from uiao_impl.adapters.terraform_adapter import TerraformAdapter
from uiao_impl.adapters.adapter_to_oscal import *
from uiao_impl.generators.sar import build_sar

adapter = TerraformAdapter({})
claims = adapter.extract_terraform_state("tests/fixtures/terraform.tfstate")
plan = json.load(open("tests/fixtures/terraform-plan.json"))
drift = adapter.consume_terraform_plan(plan)

# SAR
bundle = build_adapter_bundle("terraform", claims, control_ids=["CM-2", "CM-8"])
sar = build_sar(bundle=bundle, system_name="Training Exercise")

# POA&M
poam = build_adapter_poam("terraform", drift, ["CM-3"])

# SSP
ssp = build_adapter_ssp("terraform", claims, ["CM-8"], "Training System")
```

## 6. Module 5: Governance & Compliance (1 hour)

### 6.1 Canon-Consumer Rule

`uiao-impl/.claude/rules/canon-consumer.md`:
- No canonical artifacts in uiao-impl
- Canon changes land in uiao-core FIRST
- Tests use `tests/fixtures/` not sibling checkouts

### 6.2 Conformance CI Gate

`adapter-conformance.yml` runs on every adapter PR:
- 420/420 criteria must pass (current baseline)
- Any regression blocks merge
- Report uploaded as artifact

### 6.3 Documentation Requirements

Before an adapter reaches `status: active`:
- ATS document: Overview + Scope + Controls table
- AVS document: Validation methodology + test coverage + conformance matrix
- Fixture files committed to `tests/fixtures/`
