# UIAO FedRAMP RFC-0026 CA-7 Pathway Integration

Operational specification for how the UIAO substrate satisfies FedRAMP
RFC-0026 — the Rev5 CA-7 Continuous Monitoring Expectations update
([FedRAMP/community discussion #130](https://github.com/FedRAMP/community/discussions/130)).

This document is the operational companion to **ADR-043**. ADR-043 records
the decision and rationale; UIAO_132 records the mechanics — which adapter
emits which deliverable, which workflow fans out which cadence, and which
canon artifact answers which comment-period concern.

---

## 1. Scope

RFC-0026 introduces two paired requirements and, for each, two pathways:

| Requirement | Name | Pathway 1 (modernized) | Pathway 2 (traditional) |
|---|---|---|---|
| **RV5-CA07-VLN** | Vulnerability Reporting | VDR Balance Improvement Release | Monthly OS / DB / Web / Container / Service Config scans + monthly POA&M + annual IA scan |
| **RV5-CA07-CCM** | Collaborative Continuous Monitoring | CCM Balance Improvement Release | Monthly open ConMon meeting per the Rev5 Playbook |

Per ADR-043 D1, UIAO runs **Pathway 2** on both requirements at the
effective date (2026-06-30) and pre-wires Pathway 1 as a gated migration.

Enforcement clock (RFC-0026):
- **2026-06-30** — gradual adoption begins.
- **2026-12-31** — grace period ends.
- **2027-01-01** — enforcement begins; five-strike ladder applies.

## 2. RV5-CA07-VLN — Vulnerability Reporting (Pathway 2)

### 2.1 Deliverables and sources

| Pathway-2 deliverable | UIAO artifact that produces it | Cadence | Workflow |
|---|---|---|---|
| OS configuration scan | `scubagear` (M365 tenant scope) + `stig-compliance` slot (future, OS scope) | monthly | `conmon-scheduled.yml` |
| Database configuration scan | `stig-compliance` slot (future, DB scope) | monthly | `conmon-scheduled.yml` |
| Web application configuration scan | `stig-compliance` slot (future, web scope) | monthly | `conmon-scheduled.yml` |
| Container configuration scan | `vuln-scan` slot (future, container scope) | monthly | `conmon-scheduled.yml` |
| Service configuration scan | `scubagear` + cloud-service conformance adapters | monthly | `conmon-scheduled.yml` |
| Monthly POA&M update | Governance plane POA&M generator | monthly | `conmon-aggregate.yml` |
| Annual Independent Assessor scan | 3PAO engagement per ADR-025 D5 | annual | out-of-band, evidence pinned under `evidence/sar/` |

All six monthly scans land in `evidence/conformance/<adapter-id>/<run-id>/`;
`conmon-aggregate.yml` merges them into the POA&M CSV that is uploaded to
Connect.gov (today: manual; see §7 Open items).

### 2.2 Severity gating (self-imposed 72-hour SLA)

Per ADR-043 D2, UIAO layers a **72-hour critical-finding SLA** on top of
RFC-0026's monthly cadence. `conmon-aggregate.yml` opens a governance
issue within 24 hours of any `CRITICAL` finding and requires an acknowledged
remediation plan within 72 hours. The SLA is internal to UIAO; it is not
(yet) an RFC-0026 obligation, but it answers the timeliness gap raised by
`vedigaurav` and `lrSpriggs` in comment.

### 2.3 Connect.gov submission

The RFC requires deliverables to land in the FedRAMP POA&M Template and in
a specific Connect.gov folder. UIAO emits the CSV in the correct column
shape today; the upload step is a manual operator task documented in
`docs/docs/conmon-operator-runbook.qmd` (stub) until the automation in §7
lands.

### 2.4 Pathway-1 migration gate

When the VDR Balance Improvement Release is published, UIAO will:

1. Review the VDR deliverable contract against §2.1.
2. Amend `adapter-registry.yaml` to flip affected adapters' `pathway`
   field from `pathway-2-traditional` to `pathway-1-modernized`.
3. Retire the manual POA&M CSV submission in favor of the
   machine-readable OSCAL feed VDR defines (expected).
4. Supersede ADR-043 D2 with a successor ADR.

## 3. RV5-CA07-CCM — Collaborative ConMon (Pathway 2)

### 3.1 Meeting cadence

UIAO hosts a standing **monthly agency-open ConMon meeting** with:

- Published agenda 5 business days before the meeting.
- Attendee list open to any agency customer of record.
- Recording + minutes attached to the POA&M evidence pack for that month.
- Standing agenda items:
  1. POA&M delta since last meeting.
  2. New critical findings (per §2.2 72-hour SLA).
  3. Upstream change summary (ScubaGear pin bumps, baseline changes).
  4. Agency Q&A.
  5. Significant Change Request review, if any.

### 3.2 OSCAL evidence drop

Pre-meeting, UIAO drops three OSCAL JSON artifacts to Connect.gov:

- **SSP** — generated by the UIAO SSP generator (UIAO_101 §Platform
  Services Layer).
- **SAR** — generated by the UIAO SAR generator (driven by UIAO_110
  Drift Engine output).
- **POA&M** — generated by `conmon-aggregate.yml`.

The drop is the RFC's "adequate authorization data" answer: agency
customers consume the same feeds UIAO's own drift ledger consumes, so
there is no bespoke reporting layer to drift out of sync.

### 3.3 Agency-customer consumption interface

For agency customers with OSCAL-capable tooling, UIAO additionally
publishes a read-only signed feed under
`evidence/oscal/<yyyy-mm>/` on the same schedule. Consumers verify
provenance via the canon-anchored signatures described in UIAO_110.

### 3.4 Pathway-1 migration gate

When the CCM Balance Improvement Release is published, UIAO will:

1. Compare CCM's meeting/automation contract against §3.1–§3.3.
2. Retire any manual meeting deliverables that CCM automates.
3. Keep the monthly meeting itself unless CCM explicitly replaces it;
   the meeting is a UIAO cultural artifact, not only a compliance one.

## 4. Drift-ledger answer to BOD 25-01 / point-in-time SCuBA

Per ADR-043 D6, the `scubagear` adapter remains a point-in-time assessor.
Continuity between runs is materialized by the Drift Engine (UIAO_110):

- Every `scubagear` run produces `findings.json`.
- The Drift Engine diffs consecutive `findings.json` snapshots and emits
  `DRIFT-POLICY` findings for any policy-state change between runs.
- `DRIFT-POLICY` findings flow into the monthly POA&M and are visible to
  agency customers via the OSCAL feed in §3.2.
- BOD 25-01 scope (M365 SCuBA baselines) is therefore covered
  continuously *at the ledger layer* even though the underlying
  `scubagear` runs remain interval-class evidence.

This answers the repo owner's comment on the RFC directly: the gap is
closed in UIAO's drift fabric, not in the assessor itself.

## 5. Adapter-registry binding

Per ADR-043 D4, every conformance adapter whose `controls` list includes
`CA-7` is annotated with an advisory `fedramp-rfc-0026` block in its
`notes` body today, promoted into schema when RFC-0026 ratifies:

```
fedramp-rfc-0026:
  requirement: RV5-CA07-VLN | RV5-CA07-CCM
  pathway: pathway-2-traditional
  planned-pathway: pathway-1-modernized
  pathway-transition-trigger: "Publication of the VDR Balance Improvement Release"
  adr: ADR-043
```

Current CA-7-tagged adapters (see `canon/adapter-registry.yaml`):

| Adapter | Requirement it primarily serves | Status |
|---|---|---|
| `scubagear` | RV5-CA07-VLN (service config scan), RV5-CA07-CCM (SCuBA-baseline evidence at drop) | active (Phase 1) |
| `vuln-scan` | RV5-CA07-VLN (container + service vulnerability) | reserved |
| `stig-compliance` | RV5-CA07-VLN (OS / DB / web config scan) | reserved |
| `patch-state` | RV5-CA07-VLN (patch telemetry, secondary) | reserved |
| `intune` | RV5-CA07-VLN (endpoint-compliance telemetry) | reserved |
